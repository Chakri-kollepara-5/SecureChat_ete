How It Works

A 5-step cryptographic protocol combining asymmetric key exchange, symmetric encryption, and message authentication.

STEP 01
⚙️
Key Generation
Each user generates an RSA-2048 keypair. The public key is shared openly; the private key never leaves the user's device. A shared HMAC key is negotiated for message authentication.
RSA-OAEP · 2048-bit
STEP 02
📦
Session Key Wrap
Sender generates a fresh random 256-bit AES key for each message. This session key is encrypted (wrapped) with the recipient's RSA public key — only the recipient can unwrap it.
RSA-OAEP encryption
STEP 03
🔒
Message Encryption
The plaintext is encrypted using AES-256-GCM with a random 96-bit IV (nonce) generated per message. GCM mode provides both confidentiality and built-in integrity checking.
AES-256-GCM · 96-bit IV
STEP 04
✍️
Digital Signature
The plaintext is hashed with SHA-256, then signed using HMAC-SHA256. This proves the message came from the true sender and was not altered in transit.
HMAC-SHA256 · SHA-256
STEP 05
✅
Verify & Decrypt
Recipient first verifies the HMAC signature. Only if valid, the RSA private key unwraps the AES session key, which then decrypts the ciphertext. Tampered messages are rejected.
Verify-then-decrypt

Cryptographic Concepts

Every algorithm used in this project — what it is, why it's used, and how it's implemented.

🔑
RSA-OAEP (2048-bit)
Asymmetric / Public-Key Cryptography
RSA (Rivest–Shamir–Adleman) is a public-key cryptosystem based on the mathematical difficulty of factoring large integers. Each user has a public key (shared with anyone) and a private key (kept secret). Data encrypted with the public key can only be decrypted by the corresponding private key.

OAEP (Optimal Asymmetric Encryption Padding) adds randomized padding before encryption, preventing chosen-ciphertext attacks and making RSA semantically secure.

In SecureChat: RSA is used for key encapsulation — encrypting the AES session key, not the message itself (too slow for large data).
const keyPair = await crypto.subtle.generateKey({ name: 'RSA-OAEP', modulusLength: 2048, // key size publicExponent: new Uint8Array([1,0,1]), // e=65537 hash: 'SHA-256' }, true, ['encrypt', 'decrypt']);
🔒
AES-256-GCM
Symmetric Authenticated Encryption
AES (Advanced Encryption Standard) is the world standard for symmetric encryption. We use a 256-bit key for maximum security. GCM (Galois/Counter Mode) transforms AES into an Authenticated Encryption with Associated Data (AEAD) cipher — it provides both confidentiality AND integrity in one pass.

A fresh random 96-bit IV (Initialization Vector) is generated for every single message. This ensures that even if you send the same message twice, the ciphertext is completely different — preventing pattern analysis.

In SecureChat: AES-GCM encrypts the actual message. It's ~1000× faster than RSA for large data.
const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-bit const ct = await crypto.subtle.encrypt( { name: 'AES-GCM', iv }, aesKey, new TextEncoder().encode(plaintext) );
✍️
HMAC-SHA256
Message Authentication Code
HMAC (Hash-based Message Authentication Code) uses a shared secret key combined with a hash function (SHA-256 here) to produce an authentication tag for a message. It proves two things: authenticity (the message came from the claimed sender) and integrity (the message was not altered).

If an attacker intercepts and modifies even one byte of the message, the HMAC verification will fail — and the receiver rejects the message entirely.

In SecureChat: the receiver verifies the HMAC before decrypting — this is the "verify-then-decrypt" security model.
// Sign const sig = await crypto.subtle.sign('HMAC', hmacKey, new TextEncoder().encode(plaintext)); // Verify const ok = await crypto.subtle.verify('HMAC', hmacKey, signature, new TextEncoder().encode(received));
🔏
SHA-256
Cryptographic Hash Function
SHA-256 (Secure Hash Algorithm 256-bit) is a one-way function that maps any input to a fixed 256-bit (32-byte) output called a digest. Three properties make it cryptographically useful:

• Pre-image resistance: Cannot reverse-engineer the input from the hash.
• Collision resistance: Infeasible to find two inputs with the same hash.
• Avalanche effect: One bit change in input → ~half the bits in hash change.

In SecureChat: SHA-256 digests the plaintext message, which is then signed by HMAC. Also used internally within RSA-OAEP.
const encoded = new TextEncoder().encode(text); const hashBuf = await crypto.subtle.digest('SHA-256', encoded); // → 32-byte (256-bit) fixed-size output // → Completely different for any input change
Why Hybrid Encryption?
RSA is slow — encrypting a large file with RSA-2048 would take seconds. AES is fast but requires both parties to already share a secret key (the key distribution problem).

Hybrid encryption solves both problems: Use RSA to securely exchange a small AES key (fast RSA operation on tiny data), then use AES to encrypt the actual message at high speed. This is exactly how TLS (HTTPS), Signal, PGP/GPG, and WhatsApp work at their core.
Security Analysis

Threat model, mitigations, and known limitations of this implementation.

Threat / Attack Description Mitigation Algorithm Status
Eavesdropping Attacker intercepts message in transit and reads it All messages encrypted. Ciphertext is meaningless without the private key. AES-256-GCM ✔ Mitigated
Impersonation Attacker pretends to be Alice or Bob Only the true sender holds the HMAC signing key. Signature cannot be forged without it. HMAC-SHA256 ✔ Mitigated
Message Tampering Attacker modifies the message or ciphertext in transit HMAC verification fails if even one byte is altered. GCM authentication tag also catches ciphertext modification. HMAC-SHA256 GCM Auth Tag ✔ Mitigated
Replay Attack Attacker re-sends a previously captured valid message Every message uses a freshly generated random 96-bit IV, making each ciphertext unique even for identical plaintext. Random IV ✔ Mitigated
Brute Force Key Attacker tries all possible keys to decrypt AES-256 has 2²⁵⁶ possible keys. RSA-2048 requires factoring a 2048-bit semiprime. Both computationally infeasible. RSA-2048 AES-256 ✔ Mitigated
⚠ No PKI / CA No way to verify that Bob's public key truly belongs to Bob Real system would use X.509 certificates signed by a trusted Certificate Authority (like TLS/HTTPS does) — Known Limitation
⚠ No Forward Secrecy If the long-term RSA private key is compromised, past messages can be decrypted Real systems (Signal Protocol) use ephemeral Diffie-Hellman key exchange (Double Ratchet Algorithm) — Known Limitation
⚠ Browser Memory Private keys stored in browser memory, not hardware-secured Production systems use hardware security modules (HSM) or secure enclaves (Apple Secure Enclave, Android StrongBox) — Known Limitation
